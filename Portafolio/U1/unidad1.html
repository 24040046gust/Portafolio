Introducción al Desarrollo Web Orientado a Servicios

1. Paradigma del Desarrollo Orientado a Servicios

El desarrollo orientado a servicios representa un cambio fundamental respecto a las arquitecturas monolíticas tradicionales. En este paradigma, la lógica de negocio se descompone en unidades funcionales más pequeñas, autónomas y distribuidas (servicios) que pueden residir en diferentes servidores y comunicarse a través de redes.

Clasificación de Servicios en la Nube (Cloud Computing):
La computación en la nube ofrece servicios escalables bajo demanda, clasificados según el nivel de abstracción:

  • IaaS (Infraestructura como Servicio): Provee recursos de hardware virtualizados (servidores, redes, almacenamiento). El usuario gestiona el sistema operativo y las aplicaciones. Ej: Amazon EC2, Azure Virtual Machines.
  • PaaS (Plataforma como Servicio): Ofrece un entorno de desarrollo y despliegue en la nube, abstrae la infraestructura y permite al desarrollador enfocarse solo en el código y la base de datos. Ej: Google App Engine, Heroku.
  • SaaS (Software como Servicio): Entrega aplicaciones completas listas para el usuario final a través del navegador. El proveedor gestiona todo (infraestructura, datos, aplicación). Ej: Gmail, Salesforce, Microsoft 365.

Características de las Aplicaciones Orientadas a Servicios:

  • Granularidad: Funcionalidades divididas en tareas específicas.
  • Interoperabilidad: Capacidad de comunicarse entre diferentes lenguajes (Java, PHP, .NET) y plataformas.
  • Acoplamiento Débil: Mínima dependencia entre el consumidor y el proveedor del servicio.
  • Reusabilidad: Un mismo servicio puede ser consumido por múltiples aplicaciones.

Aplicaciones Web Híbridas (Mashups):
Un Mashup es una aplicación web que integra datos o funcionalidades de dos o más fuentes externas (APIs públicas) para crear un servicio nuevo y distinto.

  • Ejemplo Clásico: Una web inmobiliaria que toma datos de una base de datos de casas y los superpone en un mapa usando la API de Google Maps.
  • Características: Desarrollo rápido, integración de APIs de terceros (YouTube, Twitter, Maps) y enriquecimiento de la experiencia de usuario.

2. Arquitectura Orientada a Servicios (SOA)

SOA (Service-Oriented Architecture) no es una tecnología específica, sino un estilo arquitectónico para diseñar sistemas de software. Su objetivo es permitir que diferentes aplicaciones intercambien datos y participen en procesos de negocio independientemente de los sistemas operativos o lenguajes de programación subyacentes.

Principios de Diseño SOA (Manifiesto SOA):
Para que una arquitectura sea verdaderamente SOA, los servicios deben cumplir con:

  • Contrato Estandarizado: Los servicios deben tener una descripción formal (WSDL/OpenAPI) de su propósito y capacidades.
  • Abstracción: La lógica interna está oculta; solo se expone lo necesario a través del contrato.
  • Reutilización: La lógica se divide con la intención de ser agnóstica y reutilizable.
  • Autonomía: Los servicios tienen control sobre su entorno y recursos; no dependen de otros para existir.
  • Sin Estado (Statelessness): Los servicios no deben retener información de la sesión del cliente, mejorando la escalabilidad.
  • Descubrimiento: Deben ser fáciles de encontrar e interpretar (meta-datos).
  • Composibilidad: Los servicios pueden coordinarse para formar servicios más complejos.

Estándares y Protocolos Clave:
La implementación técnica de SOA se apoya en un conjunto de estándares rigurosos:

  • XML (Extensible Markup Language): Es el lenguaje base para la estructura de datos. Permite definir, validar y compartir formatos de datos entre sistemas heterogéneos.
  • SOAP (Simple Object Access Protocol): Protocolo estándar de comunicación basado en XML. Define una estructura de mensaje estricta: Envelope (sobre), Header (seguridad/metadatos) y Body (datos). Es altamente seguro (WS-Security) y transaccional.
  • WSDL (Web Services Description Language): Es un documento XML que actúa como el "manual de usuario" del servicio. Describe qué hace el servicio, dónde reside y cómo invocarlo (parámetros de entrada/salida).
  • UDDI (Universal Description, Discovery, and Integration): Actúa como un "directorio telefónico" (Yellow Pages) para servicios web. Permite a las empresas publicar sus servicios WSDL para que otros desarrolladores los encuentren.
  • REST (Representational State Transfer): A diferencia de SOAP, REST es un estilo arquitectónico más ligero y flexible. Utiliza los métodos estándar del protocolo HTTP (GET, POST, PUT, DELETE) y permite formatos de datos como JSON (más ligero que XML). Es el estándar dominante en la web moderna y aplicaciones móviles.
